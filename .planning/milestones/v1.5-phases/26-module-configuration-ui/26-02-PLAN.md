**2. Create EditorConfigSidebar.razor:**

New component at `src/OpenAnima.Core/Components/Shared/EditorConfigSidebar.razor`. Structure:

- Inject: `EditorStateService`, `IAnimaModuleConfigService`, `IAnimaContext`, `IPortRegistry`, `LanguageService`, `IStringLocalizer<SharedResources>`
- Subscribe to `_editorState.OnStateChanged`, `_animaContext.ActiveAnimaChanged`, `LangSvc.LanguageChanged` in `OnInitialized`
- Implements `IDisposable` (unsubscribe all events)
- Visibility: renders with CSS class `editor-config-sidebar visible` when `_editorState.SelectedNodeId != null`

**Top section (metadata — MODCFG-05):**
- Module name as header (from `Configuration.Nodes` matching `SelectedNodeId` → `ModuleName`)
- Close button (top-right) that calls `_editorState.ClearSelection()`
- Version info section: use `L["Editor.Config.Version"]` / value from PluginLoader or "1.0.0" default
- Description section: `L["Editor.Config.Description"]`
- Ports section: use `IPortRegistry.GetPorts(moduleName)` to list inputs/outputs (same pattern as ModuleDetailSidebar)
- Runtime status section: `L["Editor.Config.Status"]` with state from `_editorState.GetModuleState(selectedNodeId)` — show "Running" (green dot), "Error" (red dot + error message), or "Idle" (gray dot)

**Bottom section (config form — supports MODCFG-01, MODCFG-04):**
- Section header: `L["Editor.Config.Title"]`
- Load config from `_configService.GetConfig(animaId, moduleId)` into local `Dictionary<string, string> _currentConfig`
- If config is empty: show `L["Editor.Config.NoConfigFields"]` placeholder message ("This module has no configurable fields")
- If config has entries: render each key-value as a labeled input field
  - Label: the dictionary key
  - Input: text input bound to the dictionary value
  - On change: call `HandleConfigChanged(key, newValue)` which updates `_currentConfig`, runs validation, triggers auto-save debounce
- Validation (MODCFG-04): For each field, validate non-empty (basic validation). If invalid, show inline error `L["Editor.Config.ValidationError"]` below the field in red text. Invalid fields prevent saving.
- Auto-save with debounce: 500ms CancellationTokenSource pattern (same as EditorStateService.TriggerAutoSave). On successful save, show brief toast notification.

**Toast notification:**
- Simple `div` with `position: fixed; bottom: 1rem; right: 1rem` showing `L["Editor.Config.SavedToast"]`
- Show for 2 seconds then auto-hide via `Task.Delay(2000)`
- CSS class `.toast-notification` with fade animation

**Config reload on context changes:**
- When `OnStateChanged` fires: re-read `SelectedNodeId`, if changed reload config
- When `ActiveAnimaChanged` fires: reload config for current node from new Anima
- Track `_lastSelectedNodeId` and `_lastAnimaId` to detect actual changes and avoid unnecessary reloads

## Implementation Notes

**Locked decisions from CONTEXT.md:**
- Config panel on right side sidebar (same position: fixed pattern as ModuleDetailSidebar)
- Upper section: metadata (name, version, description, ports). Lower section: config form
- Single-select mode: click node opens sidebar, click another switches content
- Close only via X button (not clicking canvas — ClearSelection hides sidebar but user must click X explicitly per decision... however, ClearSelection via Escape key is existing behavior. The sidebar hides when SelectedNodeId becomes null, which is correct.)
- Auto-save after modification with debounce
- Toast on successful save
- Inline validation errors, user input preserved

**Claude's discretion (from CONTEXT.md):**
- Toast: 2-second duration, fixed bottom-right, fade animation
- Debounce: 500ms (matches existing EditorStateService.TriggerAutoSave)
- Form field layout: vertical stack with label above input, consistent with existing info-section pattern

**Pattern notes:**
- EditorConfigSidebar is editor-only. ModuleDetailSidebar is Modules page-only. No z-index conflict since they never coexist.
- SelectedNodeId becomes null on DeleteSelected (SelectedNodeIds.Clear + NotifyStateChanged), so sidebar auto-hides on node deletion.
- Config form renders generic key-value fields from whatever is stored. Phase 27 (BUILTIN) will populate actual config fields per module.

- Touch target minimum for action buttons: 44px tall on mobile (accessibility floor)

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label / badge / meta | 12px | 400 (regular) | 1.5 |
| Heading (section titles, legend) | 20px | 600 (semibold) | 1.2 |
| Display (page title h1) | 28px | 600 (semibold) | 1.2 |

Notes:
- Only two font weights are in use: 400 (regular) for body copy, labels, badges, and meta text; 600 (semibold) for all headings and display text. No intermediate weight (500) and no bold (700) are used in this phase.
- Monospace font (`var(--font-mono)`) applies at the same size/weight as the element's role — e.g. run IDs are 12px/400 mono, step counts are 14px/400 mono. No separate type scale entry needed.
- Port labels inside SVG NodeCard use 11px/400 — this is an SVG-specific exception inside the canvas, not a general typography token. Do not replicate 11px outside the SVG editor canvas.
- Nav labels: 0.875rem (14px) / 400. Nav section headers: 0.7rem (~11px) / 600 uppercase — existing sidebar pattern, unchanged.

Source: `app.css` body rule (14px/1.5), `RunLaunchPanel.razor` legend (20px/600), `RunCard.razor` .run-card-objective (14px), `MainLayout.razor.css` nav-label (0.875rem), RunStateBadge (12px — consolidated to 400).

## Color

All values reference existing CSS custom properties. Do not introduce new hex values — use `var(--token)` throughout.

| Role | CSS Variable | Resolved Hex | Usage |
|------|-------------|--------------|-------|
| Dominant (60%) | `--bg-primary` | #0f1117 | Page background, SVG editor canvas, main-content area |
| Secondary (30%) | `--surface-card` / `--surface-dark` | #1c1e2e / #161822 | Cards, sidebar, node card backgrounds, form inputs |
| Accent (10%) | `--accent-color` | #6c8cff | Reserved — see list below |
| Success semantic | `--success-color` | #4ade80 | Run state Running/Completed badge, progress bar fill (0–79%) |
| Warning semantic | `--warning-color` | #fbbf24 | Run state Paused/Interrupted badge, progress bar fill (80–100%) |
| Destructive | `--error-color` | #f87171 | Cancel Run button, stop-reason banner border, error-inline messages, error badge |
| Border | `--border-color` | #2a2d3e | All card borders, input borders, progress bar track |
| Text primary | `--text-primary` | #e4e6f0 | Body copy, headings, button labels, active nav |
| Text secondary | `--text-secondary` | #8b8fa3 | Meta text, port labels, timestamps, placeholder text |
| Text muted | `--text-muted` | #5c5f73 | Cancelled badge text, sidebar section headers |

**Accent `#6c8cff` is reserved for:**
- Active nav item left border and icon color
- Primary action buttons (`btn-primary` background)
- Input focus ring (`border-color: var(--accent-color)`)
- Logo icon background gradient base
- Node card border color when selected (via `_editorState.GetNodeBorderColor`)
- Propagation chain color grouping (Phase 47 — existing pattern, extended for workflow fan-out chains)

**Accent is NOT used for:**
- Body text, labels, or descriptive copy
- Generic hover states (use `--hover-bg` = rgba(108,140,255,0.08) instead)
- Progress bars (use `--success-color` / `--warning-color`)

**New for Phase 49 — workflow progress bar on RunCard:**
- Fill color: `var(--success-color)` when < 80% of expected steps; `var(--warning-color)` at >= 80%
- Track: `var(--border-color)`, height 4px, border-radius 2px — matches existing BudgetIndicator exactly
- Label: `var(--text-secondary)`, 12px, monospace — "X / Y nodes completed"

Source: `app.css` `:root`, `BudgetIndicator.razor`, `RunStateBadge.razor`, `StopReasonBanner.razor`, `NodeCard.razor`, `MainLayout.razor.css`.

## Focal Point

**Primary screen: Runs page during an active workflow run.**

The primary visual anchor on the Runs page during an active workflow run is the **WorkflowProgressBar** inside the active RunCard. The progress bar's fill — moving left-to-right in `--success-color` against the dark `--border-color` track — is the first element that draws the eye because it is the only animated, horizontally-expanding element on the page. It sits immediately below the run objective text and above the BudgetIndicator, placing it at the vertical center of the card. The RunStateBadge ("Running" in `--success-color`) in the card header provides the secondary anchor, confirming run state. Everything else (timestamps, workspace path, step count) reads as supporting meta at lower visual weight.

## Component Inventory

New components introduced in this phase and their design contract:

### WorkflowPresetSelector (new — in RunLaunchPanel or adjacent)

A dropdown or radio-group that lets the user pick a preset wiring configuration (e.g. "Codebase Analysis") before starting a run. The control appears between the existing Objective field and the Start Run button.

- Container: no extra card wrapper — inline inside existing `.launch-panel fieldset`
- Label: "Workflow Template" — 14px/400, `var(--text-primary)`, same label pattern as existing form fields
- Select element: full-width, `background: var(--surface-dark)`, `border: 1px solid var(--border-color)`, `border-radius: 6px`, `color: var(--text-primary)`, `font-size: 14px`, `padding: 8px 16px` — identical to existing `<input>` field styling
- Focus state: `border-color: var(--accent-color)` — matches all other inputs
- Default option text: "None — manual wiring" (pre-selected)
- Preset option text example: "Codebase Analysis (4-branch parallel)"
- When a preset is selected: show a one-line description below the select in `var(--text-secondary)` at 12px — e.g. "Scans workspace, runs 4 parallel LLM analysis branches, synthesizes a Markdown report."
- No separate loading state needed — preset list is static at startup

### WorkflowProgressBar (new — inside RunCard)

A progress indicator showing completed nodes out of total expected nodes in a workflow run. Extends the existing `BudgetIndicator` pattern.

- Position: directly below `.run-card-meta` line, above `StopReasonBanner` and `BudgetIndicator`
- Visibility condition: only render when the run has a workflow preset attached AND `TotalNodes > 0`
- Track: `height: 4px`, `background: var(--border-color)`, `border-radius: 2px` — identical to BudgetIndicator track
- Fill: `background: var(--success-color)` when `CompletedNodes / TotalNodes < 0.8`; `background: var(--warning-color)` at >= 0.8
- Fill transition: `transition: width 0.3s`
- Label: `font-size: 12px`, `color: var(--text-secondary)`, `font-family: var(--font-mono)` — format: "X / Y nodes completed"
- ARIA: `role="progressbar"`, `aria-valuenow`, `aria-valuemin="0"`, `aria-valuemax` — matches existing BudgetIndicator accessibility pattern
- When workflow completes (RunState = Completed): fill reaches 100%, `--success-color`

### JoinBarrierModule NodeCard State

The JoinBarrierModule appears as a standard NodeCard in the SVG editor. No new visual component needed. The existing NodeCard handles Running/Error/Idle states via `GetNodeStateClass()`. Specific design requirements:

- Node width: 200px (existing constant — not changed)
- Title bar: 28px height — displays "Join Barrier" as module name
- Input ports (4 total, left side): rendered in `--port-type-color` per port type — use the same `PortColors.GetHex()` pattern
- Status dot: pulsing animation (`status-pulse` class) while any branch is still awaiting — reuse existing `IsRunningState()` → `node-running` class
- No custom visual differentiation needed beyond the standard NodeCard — the module name "Join Barrier" in the title bar is sufficient

### PropagationId Chain Color (extended — existing Phase 47 infrastructure)

Phase 47 already built propagation chain color grouping in RunDetail. Phase 49 activates PropagationId tracking end-to-end. No new visual design decisions are needed — the existing `PropagationColorAssigner` and timeline chain-highlight behavior applies directly to workflow fan-out steps.

- Each workflow invocation generates one PropagationId → one color group
- Steps across the 4 parallel analysis branches share the same PropagationId if triggered from the same scan event
- The JoinBarrier step receives the same PropagationId from the synthesis trigger
- Color assignment is automatic via existing `PropagationColorAssigner` — no executor design decision needed

## Interaction States

### Workflow Run — lifecycle states visible in RunCard

| Run State | RunStateBadge | WorkflowProgressBar | Actions |
|-----------|--------------|--------------------|---------|
| Created | "Starting" — `--text-secondary` tinted | 0 / N nodes completed | none |
| Running | "Running" — `--success-color` tinted | X / N nodes completed — live via SignalR | Pause Run, Cancel Run |
| Paused | "Paused" — `--warning-color` tinted | X / N nodes completed — frozen | Resume Run, Cancel Run |
| Interrupted | "Interrupted" — `--warning-color` tinted | X / N nodes completed — frozen | Resume Run, Cancel Run |
| Completed | "Completed" — `--success-color` tinted | N / N nodes completed — 100% fill | none |
| Cancelled | "Cancelled" — `--text-muted` tinted | X / N — last value frozen | none |
| Failed | "Failed" — `--error-color` tinted | X / N — last value frozen | none |

### NodeCard — workflow execution states (existing RTIM pattern, applied to workflow)

| Module execution state | NodeCard border | Status dot | Pulse ring |
|------------------------|----------------|------------|------------|
| Idle | `--border-color` (1.2px) | `--text-muted` fill | none |
| Running | `--accent-color` (via EditorState color assignment) | `--accent-color` fill | yes — `status-pulse` animation |
| Error | `--error-color` | `--error-color` fill | no — `⚠` icon shown |
| Completed (stopped) | `--border-color` | `--success-color` fill | none |

### WorkflowPresetSelector — interaction states

- Default: border `--border-color`, background `--surface-dark`
- Focus: border `--accent-color`
- Hover: no separate hover style beyond browser default (matches existing input pattern)
- Disabled (no presets available): `opacity: 0.5`, `cursor: not-allowed` — matches `.btn:disabled` pattern

## Accessibility Requirements

These requirements apply to all new components in this phase:

- WorkflowProgressBar: `role="progressbar"`, `aria-valuenow="{X}"`, `aria-valuemin="0"`, `aria-valuemax="{N}"`, `aria-label="Workflow progress: X of N nodes completed"`
- WorkflowPresetSelector `<select>`: `id="launch-preset"`, paired `<label for="launch-preset">Workflow Template</label>`
- NodeCard status dot tooltip: existing `<title>` element pattern is used — no change needed
- Cancel Run confirm dialog: existing `ConfirmDialog` component with focus trap — no change needed
- All new buttons: `aria-label` with run ID or context suffix where the button text alone is ambiguous

## Scope Note

Phase 68 changes only the existing chat conversation surface.

UI work in this phase is limited to:
1. Explicit `memory_create`, `memory_update`, and `memory_delete` visibility inside assistant bubbles as collapsible cards.
2. A single non-expandable sedimentation summary chip attached to the same assistant bubble.
3. A distinct `ToolCategory.Memory` visual modifier that differentiates memory cards from generic workspace tool cards.

This phase does not introduce:
- A new memory-only card component
- A new page, modal, drawer, or navigation entry
- Per-node sedimentation drilldown
- Background execution affordances from Phase 69

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps inside pills, chip inner vertical inset |
| sm | 8px | Gap between metadata items, compact stack spacing, chip horizontal inset |
| md | 16px | Default card padding, assistant bubble inset, body spacing |
| lg | 24px | Separation between tool-card block and assistant markdown body |
| xl | 32px | Not introduced by this phase |
| 2xl | 48px | Not introduced by this phase |
| 3xl | 64px | Not introduced by this phase |

Exceptions:
- Existing chat bubble and tool-card shell keep inherited `12px` vertical inset (`0.75rem`) from `ChatMessage.razor.css`; Phase 68 does not retune the base bubble.
- Collapsible memory card headers must keep a minimum interactive height of `44px`.
- Folded summary rows may use `12px` gap compositions (`4px + 8px`) but no new non-multiple-of-4 values.

Typography rules for this phase:
- Memory card folded headers use `14px / 600` for the operation label and `12px / 400` for URI and one-line summary metadata.
- Expanded parameter keys, URI pills, and result text use `12px / 400`; URI values stay monospace.
- The sedimentation summary chip uses `12px / 400`; it is metadata, not a heading.
- No new weights beyond `400` and `600`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f1117` | Chat column background, expanded markdown/code recesses |
| Secondary (30%) | `#1c1e2e` | Assistant bubble surfaces, memory card container, summary chip surface |
| Accent (10%) | `#6c8cff` | Memory category stripe, memory icon, URI pill, sedimentation chip count/icon, focus ring |
| Destructive | `#f87171` | Delete memory operation badge and failed/error states only |

Accent reserved for: memory-card category stripe, memory icon, URI token or URI text accent, sedimentation summary chip icon and count, focus-visible outline on collapsible headers. Do not use accent as a full-card fill or on all generic tool cards.

Memory-specific visual treatment:
- Generic workspace tool cards remain neutral.
- `ToolCategory.Memory` cards add a `2px` left accent stripe, accent-tinted title/icon treatment, and a subtle accent wash no stronger than existing `--hover-bg`.
- `memory_delete` cards keep the same memory category shell but swap the operation badge text/border to destructive red so delete reads as destructive without turning the whole card red.
- The sedimentation summary chip uses the secondary surface plus accent text/icon; it must stay quieter than explicit memory cards.
- All new memory-specific surfaces must use existing `app.css` CSS variables. Do not introduce new hard-coded hex colors for this phase.

## Interaction Contract

### Explicit memory tool cards

- Only `memory_create`, `memory_update`, and `memory_delete` receive `ToolCategory.Memory`.
- Memory cards render in the same pre-message tool-card stack that current tool calls use, above assistant markdown content.
- Memory cards are collapsed by default on live render and on historical replay.
- Folded header order is fixed:
  1. tool status icon
  2. memory category icon
  3. operation label
  4. target URI
  5. one-line content summary when content exists
  6. chevron
- The URI must remain visible in the folded state for all three operations.
- The one-line content summary is required for create and update. Clamp it to one rendered line with ellipsis and a hard cap of `80` persisted characters before ellipsis.
- Delete cards do not fabricate a summary. They show only operation label plus URI in the folded state.
- Expanded state reuses the existing body pattern for parameter rows and result summary. No separate memory-detail layout is introduced.
- Expanded delete cards must surface the soft-delete semantics in body copy or result text.

### Sedimentation summary chip

- Render exactly one sedimentation summary chip per assistant response.
- The chip is non-expandable.
- The chip appears below the assistant markdown content in the same metadata row as the existing tool-count badge, before the generic badge when both are present.
- The chip shows count only. It never includes URI-level detail, create or update breakdown, or per-node hover content.
- If the written count is `0`, render no chip.
- The chip must feel subordinate to the conversation: secondary surface, label-sized text, and no dominant animation.

### Replay and persistence

- Memory cards and the sedimentation chip persist with the assistant message and must re-render after refresh or restart.
- Replay preserves collapsed defaults from live render: collapsed by default even for historical messages.
- Historical replay is read-only. No mutation controls, dismiss controls, or retry buttons are added to replayed memory cards.

## Accessibility

- Every collapsible memory card header remains a real `<button>` with `aria-expanded`.
- The memory category icon is decorative and should use `aria-hidden="true"`.
- URI text and operation label must remain readable without color; color is only supplementary.
- Delete cards require both destructive text and destructive color, not color alone.
- The sedimentation chip should be announced once on live insert via `aria-live="polite"` or equivalent non-intrusive status semantics.
- Keyboard focus uses the shared accent outline from `app.css`.
- Collapsible headers keep the minimum `44px` target height.

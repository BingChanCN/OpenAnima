  <action>
    **Changes to LLMModule.cs:**

    1. Add `[OutputPort("error", PortType.Text)]` attribute to class declaration (alongside existing response port).

    2. Add optional `ICrossAnimaRouter? router = null` as last constructor parameter. Store as `_router`. Because `ICrossAnimaRouter` is already registered as a singleton (Phase 28, in AnimaServiceExtensions.cs which runs before WiringServiceExtensions.cs in Program.cs), .NET DI will automatically resolve and inject it when constructing LLMModule. The `? = null` default is only for test-time direct construction where `router: null` is passed explicitly — standard DI does NOT inject null for unresolvable constructor parameters (it throws `InvalidOperationException`), but since `ICrossAnimaRouter` IS registered, this is not a concern. Add `FormatDetector` as a private field: `private readonly FormatDetector _formatDetector = new();`

    3. Rewrite `ExecuteAsync` to implement this flow:
       a. Build known service names set by querying `_configService.GetConfig(animaId, "AnimaRouteModule")` for `targetAnimaId` and `targetPortName`. If both present and `_router` is not null, look up `_router.GetPortsForAnima(targetAnimaId)` to get port descriptions.
       b. Build messages list: if known service names is non-empty, prepend a system message using `BuildSystemMessage(ports)`. Always add user message with `_pendingPrompt`.
       c. Call LLM (existing custom-client-or-global logic).
       d. If `_router` is null OR known service names is empty, publish response directly to response port (original behavior, PROMPT-04).
       e. Otherwise, call `_formatDetector.Detect(result.Content, knownServiceNames)`.
       f. If `MalformedMarkerError` is null: publish passthrough text to response port, then dispatch each route extraction by publishing to `AnimaRouteModule.port.request` then `AnimaRouteModule.port.trigger` (request FIRST, then trigger — order is critical).
       g. If `MalformedMarkerError` is non-null: enter self-correction loop. Append assistant message (LLM's bad response) + user correction message (include the error AND the correct format example). Re-call LLM. Re-detect. Up to 2 retries (MaxRetries = 2, so 3 total attempts). If still failing after max retries, publish error to `{Metadata.Name}.port.error`.

    4. Add private method `BuildSystemMessage(IReadOnlyList<PortRegistration> ports)` that returns the system prompt string:
       ```
       You have access to the following services. To use a service, include a routing marker in your response.

       Available services:
       - {portName}: {description}

       Routing marker format:
       <route service="{portName}">your request to the service</route>

       You may include multiple routing markers in a single response. Any text outside markers is delivered to the user normally.
       ```
       Per user decision: no token budget cap. Inject all services (PROMPT-02).

    5. Add private method `BuildKnownServiceNames(string animaId)` returning `HashSet<string>(StringComparer.OrdinalIgnoreCase)` — queries config for AnimaRouteModule's targetPortName. Returns empty set if no route config found or router is null.

    6. Add private method `DispatchRoutesAsync(IReadOnlyList<RouteExtraction> routes, CancellationToken ct)` — for each route, publishes to `AnimaRouteModule.port.request` (payload) then `AnimaRouteModule.port.trigger` ("trigger").

    **No changes to WiringServiceExtensions.cs.** ICrossAnimaRouter is already registered as a singleton in AnimaServiceExtensions.cs. LLMModule is registered via `services.AddSingleton<LLMModule>()` in WiringServiceExtensions.cs. .NET DI auto-resolves all constructor parameters from the container. Since ICrossAnimaRouter IS registered, DI will inject it. No registration changes needed.

    **Changes to existing ModuleTests.cs:**
    Update the two existing LLMModule tests to pass the new optional `router: null` parameter explicitly to maintain test clarity. The tests should continue to pass without changes (optional param = null), but explicitly passing null documents the intent.

    **New test file `PromptInjectionIntegrationTests.cs`:**
    Create integration test class with:
    - `CapturingFakeLlmService` that records the messages list passed to CompleteAsync and returns configurable responses (can return different responses on subsequent calls for retry testing)
    - Tests for all 7 behaviors listed above
    - Use real EventBus, real FormatDetector, fake LLMService, NullAnimaModuleConfigService (extended to support pre-set configs), real AnimaContext
    - Tag with `[Trait("Category", "Integration")]`

    **CRITICAL anti-patterns to avoid (from RESEARCH.md):**
    - Do NOT discover routes at initialization time — query GetConfig on every ExecuteAsync call
    - Do NOT modify `_pendingPrompt` during retry — build new messages list with appended correction turns
    - Do NOT publish trigger before request — always request FIRST
    - Correction message MUST include the expected format example, not just the error reason
    - Keep correction turns ephemeral (not persisted to ChatContextManager)

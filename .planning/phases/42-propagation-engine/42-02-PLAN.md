  <action>
Modify FixedTextModule to be trigger-port-driven instead of .execute-event-driven. Use HttpRequestModule's trigger pattern as reference.

1. ADD attribute: `[InputPort("trigger", PortType.Trigger)]` to the class declaration (alongside existing `[OutputPort("output", PortType.Text)]`).

2. MODIFY `InitializeAsync`: Replace the `.execute` subscription with a trigger port subscription. Change from:
   ```csharp
   var sub = _eventBus.Subscribe<string>(
       $"{Metadata.Name}.execute",
       async (evt, ct) => await ExecuteAsync(ct));
   ```
   To:
   ```csharp
   var sub = _eventBus.Subscribe<DateTime>(
       $"{Metadata.Name}.port.trigger",
       async (evt, ct) => await ExecuteAsync(ct));
   ```
   Note: PortType.Trigger uses `DateTime` payload (matching HeartbeatModule's tick output and WiringEngine's CreateRoutingSubscription which subscribes Trigger ports as `Subscribe<DateTime>`).

3. REMOVE the `[StatelessModule]` attribute from the class. FixedTextModule reads config on every execution (IModuleConfig.GetConfig) which depends on the active Anima — it is not truly stateless in the propagation model. The StatelessModule attribute was only relevant for the old stateless dispatch fork which is now removed.

4. KEEP: `ExecuteAsync` method body unchanged — it reads config, interpolates template, publishes to output port. This is correct behavior for trigger-driven execution.

5. KEEP: `ShutdownAsync`, `GetState`, `GetLastError` unchanged.

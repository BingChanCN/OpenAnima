**Modify src/OpenAnima.Core/Modules/ChatInputModule.cs:**
- Add `[StatelessModule]` attribute (it's classified as stateless per CONTEXT.md)
- Add `using OpenAnima.Core.Channels;`
- Add a field: `private ActivityChannelHost? _channelHost;`
- Add method: `public void SetChannelHost(ActivityChannelHost channelHost) => _channelHost = channelHost;`
- Modify `SendMessageAsync`: If `_channelHost != null`, call `_channelHost.EnqueueChat(new ChatWorkItem(message, ct))` and return. Otherwise fall back to existing direct EventBus publish (backward compat for tests without full runtime).
- AnimaRuntime constructor already handles the callback wiring — the onChat callback publishes the event to EventBus. The ChatInputModule just enqueues and returns.
- NOTE: AnimaRuntime needs to set the channel host on ChatInputModule. Since ChatInputModule is registered as a DI singleton and AnimaRuntime doesn't currently have direct access to it, the wiring happens via the service layer. However, looking at the codebase pattern, ChatInputModule is resolved from DI and doesn't have a reference to AnimaRuntime. The simplest approach: keep `SendMessageAsync` publishing directly to EventBus (its existing behavior), and have the **caller** (the Blazor UI component or service calling `SendMessageAsync`) route through the channel host. But per CONTEXT.md: "ChatInputModule calls channelHost.EnqueueChat(message)". So add the setter and let the initialization service wire it.
- Update `AnimaRuntime` constructor (from Task 1 changes) to NOT hardcode ChatInputModule event name in onChat callback — instead, the onChat callback should call the existing `ChatInputModule.SendMessageAsync` or simply `EventBus.PublishAsync` with the ChatWorkItem data. The cleanest approach: onChat callback publishes to EventBus as designed in Task 1. ChatInputModule.SendMessageAsync enqueues to channel if available, which then publishes to EventBus via the callback. This creates a loop. **Resolution**: ChatInputModule.SendMessageAsync should directly enqueue to channel host (bypassing EventBus publish on the input side). The channel host's onChat callback then publishes to EventBus. No loop because ChatInputModule writes to channel, channel consumer publishes to EventBus. ChatInputModule never publishes to EventBus directly when channel host is present.
